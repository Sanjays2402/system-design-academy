Design Distributed Lock Service

Coordinate exclusive work using leases and fencing rather than trusting process ownership.

SDE245 min100K lock ops/s · lease precision in seconds · 5-node quorum
Clarify before drawing

Confirm scope, traffic shape, latency and durability targets, regional requirements, and which operation must remain correct during partial failure.

01 · REQUIREMENTSDefine the contract

Functional

User outcomes

  • Users can initiate and inspect the core distributed lock service workflow.
  • Operators can configure policy, recover failures, and audit important state changes.
  • Clients can retry safely and observe a stable result.
  • Administrators can disable or repair problematic resources.
Non-functional

System guarantees

  • Meet the stated latency and availability target.
  • Preserve the primary correctness invariant under concurrency.
  • Scale horizontally without one global hot partition.
  • Expose enough telemetry to detect lag, saturation, and loss.

08 · PRODUCTIONFailures, security, and observability

Failure posture

  • Timeout every dependency.
  • Retry only safe operations with jitter.
  • Use circuit breakers and admission control.
  • Reconcile durable truth after recovery.

Security posture

  • Authenticate at ingress; authorize per resource.
  • Encrypt in transit and at rest.
  • Redact sensitive logs.
  • Audit privileged mutations.

Golden signals

  • Rate, errors, and latency percentiles.
  • Queue lag and oldest-item age.
  • Cache hit ratio and shard skew.
  • Replica lag and reconciliation gaps.

03B · DATARecords, indexes, and consistency boundaries

Authoritative state

Use a replicated consensus log, lease/session records, monotonic fencing counters, and watches derived from committed state.

Keep the record narrow on the hot path. Large payloads, histories, and analytics belong in systems designed for sequential writes or object storage.

Partition and consistency

Store coordination state in a small consensus group and partition independent lock namespaces only after proving one group is insufficient.

Use strong or conditional consistency for ownership and state transitions. Read models, dashboards, and derived search/index views may lag if the UI communicates freshness.

Index discipline

  • Every index must correspond to a concrete query; extra indexes multiply write cost and migration risk.
  • Cursor pagination should encode a stable sort key plus unique ID; offsets become slow and unstable as data changes.
  • Tombstones preserve delete/disable semantics during replication and cache invalidation; purge later under retention policy.
  • Store schema/event versions so rolling deployments and replays can understand old records.

05B · ALGORITHMOnline decision and idempotent mutation

Acquire through consensus, return a lease and fencing token, renew before expiry, and require the protected resource to reject older tokens.

handle(command):
    identity = authenticate(command.credentials)
    validate(command.payload)
    prior = idempotency_store.get(identity, command.key)
    if prior: return prior.result

    route = partition_map.owner(command.resource_id)
    current = authoritative_store.read(route)
    next_state, side_effects = domain.apply(current, command)
    authoritative_store.compare_and_set(current.version, next_state)
    idempotency_store.commit(command.key, next_state.result)
    event_log.append(side_effects)
    return next_state.result
Exactly-once business effect

Networks only give retries and uncertainty. Combine idempotency records, conditional writes, immutable event IDs, and reconciliation to make the user-visible effect occur once.

06B · PERFORMANCECaching, hot keys, and backpressure

Cache policy

Use cache-aside for read-heavy immutable or slowly changing records. Version keys or publish invalidations after commit. Add TTL jitter and short negative caching.

Hot-key defense

Detect per-key skew, add process-local and replicated reads, coalesce misses, salt only where ordering permits, and isolate abusive tenants.

Backpressure

Bound every queue and buffer. Shed low-priority work, return retry timing, and scale from lag/oldest-item age—not CPU alone.

A cache is not a correctness layer. The authoritative store and version rules determine truth; cached entries are disposable projections that must be rebuildable.

08C · EVOLUTIONMulti-region, migrations, and rollout

Start with multi-zone

Replicate within one region, automate failover, and prove backups and reconciliation before accepting cross-region write complexity.

Add global reads or stateless edges

Route latency-sensitive reads near users while retaining one mutation owner when correctness coordination is expensive.

Introduce regional ownership

Assign each aggregate or partition a home region. Move ownership through an epoch/fencing transition rather than simultaneous writers.

Migrate safely

Use expand/contract schemas, dual-read verification, backfills with checkpoints, shadow traffic, canaries, and reversible cutovers.

08D · SECURITYThreat model and abuse controls

Authenticate lock namespaces, authorize protected resources, cap lease counts, and audit forced releases or operator intervention.

Trust boundaries

  • External client to edge: authentication, validation, rate limits, payload caps.
  • Service to service: workload identity, least privilege, encryption, deadline propagation.
  • Control plane to data plane: signed/versioned configuration and audited writers.
  • Operator access: just-in-time privilege and immutable audit logs.

Abuse and privacy

  • Quotas by tenant, actor, resource, and network—not one spoofable dimension.
  • Minimize retained personal data and enforce deletion/retention workflows.
  • Scan or sandbox untrusted content and prevent SSRF in fetch/preview features.
  • Alert on enumeration, anomalous fan-out, repeated conflicts, and privilege changes.

09C · ANSWER DEPTHFollow-up answers, not prompts

Why are fencing tokens required?

Position. Back locks with a consensus system and represent ownership as a lease that expires without renewal.

Mechanism. Client opens an authenticated session and proposes acquire to the consensus leader.

Failure test. Old owner resumes after long pause; explain the retry identity, durable checkpoint, and reconciliation signal.

Trade-off. The stronger coordination choice protects the invariant but adds latency and lowers availability during partitions; keep derived reads asynchronous.

What if a client pauses past its lease?

Position. Issue monotonically increasing fencing tokens; the resource must reject operations from stale owners.

Mechanism. Committed acquire increments fencing counter and returns lease plus token.

Failure test. Leader changes during renewal; explain the retry identity, durable checkpoint, and reconciliation signal.

Trade-off. The materialized/proactive path improves read latency but costs write amplification, repair tooling, and lag monitoring.

How do fair locks work?

Position. Never rely on wall-clock equality or a client releasing correctly; design for pauses and partitions.

Mechanism. Protected resource requires token on every mutation and rejects values below its maximum.

Failure test. Client reaches resource after lease loss; explain the retry identity, durable checkpoint, and reconciliation signal.

Trade-off. The simpler centralized design is an excellent first version; partition only when measured throughput, isolation, or failure-domain needs justify it.

How does leader election differ?

Position. Back locks with a consensus system and represent ownership as a lease that expires without renewal.

Mechanism. Renewals extend the lease only after quorum commit; client stops work before uncertainty margin.

Failure test. Watch misses compaction window; explain the retry identity, durable checkpoint, and reconciliation signal.

Trade-off. More aggressive caching or batching improves throughput but increases staleness and uncertain work; version every projection and bound the window.

When should you avoid distributed locks?

Position. Issue monotonically increasing fencing tokens; the resource must reject operations from stale owners.

Mechanism. Expiry/release commits a new revision and notifies ordered watchers.

Failure test. Administrator force-releases active lock; explain the retry identity, durable checkpoint, and reconciliation signal.

Trade-off. Operational simplicity is a feature: prefer one explicit ownership model over clever multi-writer behavior unless product requirements demand it.
